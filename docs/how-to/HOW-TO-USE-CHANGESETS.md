# How to Changesets - Versioning & Changelog

> **Tool:** `@changesets/cli` v2.29.8 | **Updated:** 2026-04-28
> **Docs:** <https://github.com/changesets/changesets>
>
> **2026-04-28 changes:** Custom changelog formatter (Conv-Commits → Keep-a-Changelog routing), CI-enforced lint on `.changeset/*.md`, auto-GitHub-Release on `v*` tag push. See sections "Body Format" + "Lint" + "Auto-GitHub-Release" below.

---

## What is Changesets?

Changesets captures **what changed** and **which semver bump is needed** — at the time of the change, not at release time. A changeset is a Markdown file with YAML frontmatter in `.changeset/`.

**Why:** Git commit messages are unsuitable for changelogs. Changesets decouple the versioning decision from the release process.

## Files

| File/Directory                         | Purpose                                                                                             |
| -------------------------------------- | --------------------------------------------------------------------------------------------------- |
| `.changeset/config.json`               | Configuration (Fixed Versioning, baseBranch, custom formatter wiring)                               |
| `.changeset/README.md`                 | Contributor quick-reference (German) — prefix rules + lint                                          |
| `.changeset/*.md`                      | Pending changesets (temporary, consumed during version bump)                                        |
| `.changeset/changeset-formatter.cjs`   | Custom Conv-Commits → Keep-a-Changelog formatter (added 2026-04-28)                                 |
| `scripts/lint-changesets.mjs`          | CI lint for `.changeset/*.md` quality (added 2026-04-28)                                            |
| `scripts/sync-root-version.mjs`        | Syncs root `package.json.version` + README badge after `changeset version`                          |
| `scripts/aggregate-changelog.mjs`      | Merges per-package CHANGELOGs into root `CHANGELOG.md` (consumed by UI), re-sections by KaC headers |
| `.github/workflows/changeset-lint.yml` | CI workflow that runs the lint script on every PR touching `.changeset/**` (added 2026-04-28)       |
| `.github/workflows/docker-build.yml`   | Auto-creates GitHub Release on `v*` tag push from CHANGELOG.md section (added 2026-04-28)           |
| `backend/CHANGELOG.md`                 | Generated by Changesets — per-package history (backend)                                             |
| `frontend/CHANGELOG.md`                | Generated by Changesets — per-package history (frontend)                                            |
| `shared/CHANGELOG.md`                  | Generated by Changesets — per-package history (`@assixx/shared`)                                    |
| `CHANGELOG.md` (root)                  | **Generated artefact** — aggregation of the three above, shown at `/versioninfo`. Never hand-edit.  |

> **All CHANGELOG.md files are committed** (not gitignored). Docker-Build &
> Vite `?raw` import for `/versioninfo` depend on the root file existing at
> checkout time. On merge conflicts: **re-run `node scripts/aggregate-changelog.mjs`**
> instead of manual merging.

> **Docker-Build dependencies (two files must stay in lockstep):**
>
> 1. `docker/Dockerfile.frontend` has a `COPY CHANGELOG.md ./` step (between
>    the shared-build and `COPY frontend/`). The Vite build reads the file
>    via `?raw` import — if missing, the build fails with
>    `[UNRESOLVED_IMPORT] Could not resolve '../../../../../CHANGELOG.md?raw'`.
> 2. `.dockerignore` has an explicit `!CHANGELOG.md` exception. The blanket
>    `*.md` rule would otherwise strip the root CHANGELOG from the build
>    context, causing
>    `failed to compute cache key: "/CHANGELOG.md": not found` at the
>    `COPY CHANGELOG.md ./` step. The per-package CHANGELOGs
>    (`backend/`, `frontend/`, `shared/`) are unaffected because `*.md` uses
>    filepath.Match semantics (no `/` matching) and only excludes top-level
>    files.
>
> Any future restructure that moves the root CHANGELOG must update both
> files in lockstep.

```bash
# 1. Create changeset (after feature work)
pnpm changeset

# 2. Lint locally (CI runs this too — fails PR on violations)
pnpm changeset:lint

# 3. Bump versions (release manager, before tagging)
pnpm changeset:version

# 4. Create git tag
pnpm changeset:tag

# 5. Push with tags — GitHub Action auto-creates the GH Release

```

## Body Format (mandatory since 2026-04-28)

Every changeset's first non-empty body line MUST start with a Conventional-Commits prefix. The custom formatter (`.changeset/changeset-formatter.cjs`) does FOUR things in one pass:

1. **Routes** to a Keep-a-Changelog section in the root CHANGELOG (`feat:` → Added, `fix:` → Fixed, etc.)
2. **Strips** the Conv-Commits prefix from the rendered body
3. **Converts** the first verb from imperative to past-tense (`add` → `Added`, `bump` → `Bumped`, `fix` → `Fixed`, …) — claude-code-style action-verb-led bullets without authors needing to write past-tense themselves
4. **Appends a PR link** if `(#NNN)` appears anywhere in the body

You keep writing what you always wrote (`feat: add foo bar`); the formatter polishes it to `Added foo bar` under `### Added`.

### Verb-tense lookup table (formatter + backfill)

`add → Added`, `bump → Bumped`, `cache → Cached`, `cleanup → Cleaned up`, `clarify → Clarified`, `deprecate → Deprecated`, `disable → Disabled`, `document → Documented`, `drop → Dropped`, `enable → Enabled`, `extract → Extracted`, `fix → Fixed`, `improve → Improved`, `introduce → Introduced`, `move → Moved`, `pin → Pinned`, `refactor → Refactored`, `refresh → Refreshed`, `remove → Removed`, `rename → Renamed`, `replace → Replaced`, `revert → Reverted`, `switch → Switched`, `update → Updated`, `use → Used`. Any verb not in the table → first letter is just capitalized (`fix: cross-origin escrow regression` → `Cross-origin escrow regression`).

### Prefix → KaC Section

| Prefix       | Routes to       | Example                                    |
| ------------ | --------------- | ------------------------------------------ |
| `feat:`      | **Added**       | `feat: add inventory list management`      |
| `fix:`       | **Fixed**       | `fix(auth): cross-origin escrow bootstrap` |
| `refactor:`  | **Changed**     | `refactor: extract permission service`     |
| `perf:`      | **Performance** | `perf: cache org-scope per request`        |
| `docs:`      | **Docs**        | `docs: clarify ADR-019 RLS-strict mode`    |
| `chore:`     | **Maintenance** | `chore: bump grafana to v13`               |
| `style:`     | **Maintenance** | `style: align design-tokens spacing`       |
| `test:`      | **Maintenance** | `test: add KVP delegation API coverage`    |
| `ci:`        | **Maintenance** | `ci: pin trivy to v0.69.3`                 |
| `build:`     | **Maintenance** | `build: switch to pnpm deploy pattern`     |
| `<prefix>!:` | **Breaking**    | `refactor!: drop V1 API surface`           |

`(scope)` is optional and preserved in the prefix detection but stripped from the rendered output. Bullets without a prefix fall through to a `### Other` section — only legacy entries (v0.1.0–v0.4.13) live there.

### Examples

**Good:**

```md
---
'assixx-backend': minor
---

feat: add inventory list management

Tenants can now maintain custom equipment inventories with tags,
categories, and per-asset photos. Closes ADR-040.
```

```md
---
'assixx-backend': patch
---

fix(auth): cross-origin first-escrow bootstrap (#219)

Apex login now creates the first escrow blob automatically during the
subdomain handoff. Closes ADR-022 §"New-user scenario".
```

**Bad (blocked by lint):**

```md
---
'assixx-backend': patch
---

chore(docker): backend production image switches to pnpm deploy pattern (closes ADR-027 §3 deferred); cuts assixx-backend:prod image size from 1.27 GB to 614 MB (-52%) by mirroring Dockerfile.frontend
```

→ First line 167 chars (max 100), 6 distinct facts in one sentence. Fix: split into 3 changesets — image-size cut, devDep moves, ARCHITECTURE map fix — one fact per file.

## Lint

`scripts/lint-changesets.mjs` (run via `pnpm changeset:lint`) blocks any `.changeset/*.md` that violates:

| #   | Rule                                                                                      |
| --- | ----------------------------------------------------------------------------------------- |
| 1   | Front-matter present (`---` block) with ≥1 package + valid bump                           |
| 2   | First non-empty body line has a Conv-Commits prefix                                       |
| 3   | First line ≤ **100 characters** total                                                     |
| 4   | Body has at least one non-empty line                                                      |
| 5   | ≤ **3** top-level bullets (`- ` at column 0) — split bigger work into multiple changesets |

CI runs this automatically via `.github/workflows/changeset-lint.yml` on every PR or push touching `.changeset/**`. Single-purpose workflow — runs in <10 seconds, no pnpm install.

To test locally with fixtures: `node scripts/lint-changesets.mjs --dir /tmp/my-test-fixtures`.

## ISO Release Dates (since 2026-04-28)

The aggregator stamps each version header with its ISO release date:

```
## 0.4.13 — 2026-04-28
```

Resolution order in `scripts/aggregate-changelog.mjs::getReleaseDate()`:

1. `git log -1 --format=%aI v<version>` — pulls the matching tag's commit date (real historical release date)
2. Fallback: `new Date().toISOString().split('T')[0]` — today's date when the tag doesn't exist yet (in-progress release inside `pnpm changeset:version`)

So all PAST versions get their actual git-tag dates. The CURRENT version (being released) gets today's date — which matches the tag-date within minutes once `pnpm changeset:tag` runs. Edge case: if `git` is unavailable (tarball checkout, missing binary), all versions get today's date. Acceptable degradation.

## Auto-GitHub-Release (since 2026-04-28)

`docker-build.yml` now creates a GitHub Release on every `v*` tag push, after the Docker build and Trivy CRITICAL-CVE scan have passed. Body extraction:

```bash
# What the workflow does internally
VERSION="${GITHUB_REF_NAME#v}"
awk -v v="$VERSION" '/^## /{ if (p) exit; if ($0 == "## " v) { p=1; next } } p { print }' \
    CHANGELOG.md > release-notes.md
```

If the version is missing from `CHANGELOG.md` (you forgot `pnpm changeset:version` before tagging), the workflow fails loud — no empty release. Pre-1.0.0 tags (`v0.x.y`) are marked as pre-release; post-1.0.0 are full releases.

Closed the gap: 28/29 historical tags had no GitHub Release before this job existed (only `v0.4.4` was created manually). From `v0.4.14` onward, `git push --follow-tags` triggers the release automatically.

## Detailed Workflow

### Step 1: Create a Changeset

After a relevant change (feature, bugfix, breaking change):

```bash
pnpm changeset
```

Interactive prompt:

1. **Which packages?** — Not applicable (Fixed Group — all are bumped together)
2. **Semver bump?** — `patch` / `minor` / `major`
3. **Description** — Becomes the CHANGELOG entry (Markdown allowed)

This creates a file like `.changeset/brave-lions-roar.md`:

```markdown
---
'assixx-backend': minor
---

TPM module: maintenance plans and Kamishibai board implemented
```

**This file is committed with the PR.**

### Step 2: Bump Versions (Prepare Release)

```bash
pnpm changeset:version
```

What happens:

1. `changeset version` runs — all `.changeset/*.md` files are **consumed** (deleted)
2. `package.json` versions are bumped (all 3 workspace packages in sync via Fixed Group)
3. Per-package `CHANGELOG.md` files (`backend/`, `frontend/`, `shared/`) are generated/updated from changeset descriptions
4. `scripts/sync-root-version.mjs` runs — syncs root `package.json.version` + README badge
5. `scripts/aggregate-changelog.mjs` runs — merges the three per-package CHANGELOGs into the root `CHANGELOG.md` (dedup + dependency-noise strip), which is the file rendered at `/versioninfo`

Steps 4 and 5 are chained inline in the `changeset:version` script — no manual action needed.

**Commit the result** (includes per-package CHANGELOGs, root CHANGELOG, and all `package.json` files).

### Step 3: Create Git Tag

```bash
pnpm changeset:tag
```

Creates a single annotated git tag like `v0.4.5` (reads version from `package.json`).

> **Note:** We do **not** use `changeset tag` (which creates per-package tags like `assixx-backend@0.4.5`). Assixx is a single deployable product with Fixed Group — one `v`-tag is sufficient. The `v*`-tag triggers the Docker build workflow.

### Step 4: Push

```bash
git push --follow-tags
```

The `v*`-tag triggers the Docker build in `.github/workflows/docker-build.yml`.

## Semver Rules

| Bump    | When                                     | Example           |
| ------- | ---------------------------------------- | ----------------- |
| `patch` | Bugfix, small improvement, no API change | `0.3.2` → `0.3.3` |
| `minor` | New feature, backwards compatible        | `0.3.2` → `0.4.0` |
| `major` | Breaking change, migration required      | `0.3.2` → `1.0.0` |

**When uncertain:** `minor` for features, `patch` for everything else.

## Configuration

File: `.changeset/config.json`

```json
{
  "changelog": "@changesets/cli/changelog",
  "commit": false,
  "fixed": [["assixx-backend", "assixx-frontend", "@assixx/shared"]],
  "access": "restricted",
  "baseBranch": "main",
  "updateInternalDependencies": "patch",
  "privatePackages": { "version": true, "tag": true }
}
```

| Option            | Value                      | Meaning                                         |
| ----------------- | -------------------------- | ----------------------------------------------- |
| `fixed`           | All 3 workspace packages   | Lockstep — always same version                  |
| `commit`          | `false`                    | Changesets does not auto-commit                 |
| `access`          | `restricted`               | No npm publish (private project)                |
| `baseBranch`      | `main`                     | Base branch for comparisons                     |
| `privatePackages` | `version: true, tag: true` | Private packages are still versioned and tagged |

## Common Scenarios

### Feature Branch with Multiple Changes

```bash
# First change
git commit -m "feat(tpm): add maintenance plan service"
pnpm changeset
# → minor, "TPM: maintenance plan service implemented"

# Second change (same branch)
git commit -m "feat(tpm): add card execution"
pnpm changeset
# → minor, "TPM: card execution implemented"

# Both changesets are aggregated at the next `changeset version`
# → Result: ONE minor bump with both entries in the CHANGELOG
```

### Bugfix

```bash
git commit -m "fix(auth): token refresh race condition"
pnpm changeset
# → patch, "Auth: fixed race condition in token refresh"
```

### No Version-Worthy Change

For changes that don't justify a release (docs, CI, refactoring without external impact):

```bash
pnpm changeset --empty
# Or: Simply don't create a changeset
```

### Edit a Changeset After Creation

Changesets are normal Markdown files. Just open and edit:

```bash
# Find the file
ls .changeset/*.md

# Edit (change description, adjust bump level)
code .changeset/brave-lions-roar.md
```

### Multiple Changesets Aggregation

When multiple changesets have the same bump type, the **highest** wins:

- 3x `patch` → `patch`
- 2x `patch` + 1x `minor` → `minor`
- 1x `minor` + 1x `major` → `major`

All descriptions become separate entries in the CHANGELOG.

## Fixed Group — What Does It Mean?

All 3 workspace packages (`assixx-backend`, `assixx-frontend`, `@assixx/shared`) are in a **Fixed Group**. This means:

- A changeset for **one** package bumps **all three** to the same version
- Versions never drift apart
- The root package (`assixx`) is automatically synced via `scripts/sync-root-version.mjs`

**Why:** Assixx is a deployable product, not a library collection. One version = one release state.

## Troubleshooting

### "Some packages have been changed but no changesets were found"

```bash
# Expected error when changes exist but no changeset
# Solution: Create a changeset or empty changeset
pnpm changeset
# or
pnpm changeset --empty
```

### "tag already exists"

```bash
# Tag was already created (e.g. from a previous run)
# Solution: Delete locally and recreate
git tag -d v0.4.5
pnpm changeset:tag
```

### CHANGELOG.md Looks Wrong

- Changeset files are editable — adjust description before `changeset version`
- After `changeset version`, `CHANGELOG.md` can be edited directly (before committing)

### Version Mismatch

```bash
# Check all versions
node -e "
import {readFileSync} from 'fs';
['package.json','backend/package.json','frontend/package.json','shared/package.json']
  .forEach(f => {
    const p = JSON.parse(readFileSync(f, 'utf8'));
    process.stdout.write(p.name.padEnd(20) + ' -> ' + p.version + '\n');
  });
"
```

### js-yaml Error

The `pnpm.overrides` in `package.json` contains a restricted override for `js-yaml`:

```json
"js-yaml@>=4.0.0 <4.1.1": ">=4.1.1"
```

This override must **not** be expanded to `"js-yaml@<4.1.1"` — that breaks `read-yaml-file` (changeset dependency), which needs js-yaml 3.x with `safeLoad`.

## Architecture Decision

| Aspect              | Decision                                      | Reasoning                                                            |
| ------------------- | --------------------------------------------- | -------------------------------------------------------------------- |
| Tool                | Changesets (not semantic-release, not lerna)  | KISS, Markdown-based, pnpm-native                                    |
| Versioning Strategy | Fixed Group                                   | One product = one version                                            |
| npm Publish         | No (`access: restricted`)                     | Private SaaS, not a public package                                   |
| Auto-Commit         | No (`commit: false`)                          | Manual control over commits                                          |
| CHANGELOG Format    | Custom (`.changeset/changeset-formatter.cjs`) | Conv-Commits → Keep-a-Changelog routing + PR-link enrichment         |
| Bullet style        | Action-verb-led past-tense (`Added foo`)      | Inspired by claude-code; auto-converted from imperative by formatter |
| Release dates       | ISO `## VERSION — YYYY-MM-DD` from git tag    | Tenant-admin "when did this ship?" signal; today fallback if no tag  |
| Body Discipline     | CI lint (`scripts/lint-changesets.mjs`)       | Forcing function — prevents 0.4.13-style run-on bullets at source    |
| Root Sync           | `scripts/sync-root-version.mjs`               | Changesets only manages workspace packages                           |
| Root CHANGELOG      | `scripts/aggregate-changelog.mjs`             | Single source for `/versioninfo`; KaC re-sectioning by tag           |
| Tagging             | Custom `git tag -a` (not `changeset tag`)     | Single `v`-tag, no per-package tag noise                             |
| GitHub Release      | Auto-created on `v*` tag push                 | `softprops/action-gh-release@v2` step in `docker-build.yml`          |
